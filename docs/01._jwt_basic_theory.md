# JWT 기본 이론 및 Access Token / Refresh Token 개념 정리

## 1. 문서 목적

본 문서는 JWT 기반 인증 구조에서 Access Token과 Refresh Token의 기본 개념을 정리하기 위한 문서입니다.

본 문서의 목적은 다음과 같습니다.

```text
1. JWT가 무엇인지 설명
2. Access Token의 역할 설명
3. Refresh Token의 역할 설명
4. Access Token과 Refresh Token의 차이 설명
5. Refresh Token이 클라이언트와 서버에서 어떻게 다뤄지는지 설명
6. Access Token 만료 후 재발급 흐름 설명
7. 잘못 이해하기 쉬운 지점 정리
```

본 문서는 특정 시스템의 실제 구현을 단정하기 위한 문서가 아니라, JWT 인증 구조를 이해하기 위한 기본 설명 문서입니다.

---

## 2. JWT란 무엇인가

JWT는 JSON Web Token의 약자입니다.

일반적으로 사용자의 인증 상태나 권한 정보를 담아 서버와 클라이언트 사이에서 전달하는 토큰 형식입니다.

JWT는 보통 다음 세 부분으로 구성됩니다.

```text
Header.Payload.Signature
```

예시 형태:

```text
xxxxx.yyyyy.zzzzz
```

각 영역의 역할은 다음과 같습니다.

| 구분        | 설명                           |
| --------- | ---------------------------- |
| Header    | 토큰 타입, 서명 알고리즘 정보            |
| Payload   | 사용자 식별자, 만료시간, 권한 등 claim 정보 |
| Signature | 서버가 토큰 위변조 여부를 검증하기 위한 서명    |

JWT는 암호화된 값처럼 보이지만, 일반적인 JWT는 **암호화가 아니라 서명된 값**입니다.

즉, Payload는 Base64URL로 인코딩되어 있을 뿐이며, 누구나 디코딩해서 내용을 볼 수 있습니다.

따라서 JWT Payload에는 비밀번호, 주민번호, 카드번호, 민감 개인정보 같은 값을 넣으면 안 됩니다.

---

## 3. JWT의 핵심은 서명 검증이다

JWT에서 중요한 것은 “토큰 문자열이 존재한다”가 아닙니다.

중요한 것은 서버가 다음 항목을 검증하는 것입니다.

```text
1. 이 토큰이 서버가 발급한 토큰인가?
2. 중간에 위변조되지 않았는가?
3. 만료시간이 지나지 않았는가?
4. 이 토큰이 어떤 용도의 토큰인가?
5. 이 사용자가 아직 유효한 상태인가?
6. 서버 저장소 기준으로 폐기되지 않은 토큰인가?
```

JWT는 서명 검증이 실패하면 신뢰할 수 없습니다.

또한 서명 검증이 성공하더라도 만료시간, 토큰 타입, 계정 상태, 폐기 여부 등을 추가로 확인해야 할 수 있습니다.

---

## 4. Access Token이란 무엇인가

Access Token은 API 접근에 사용하는 토큰입니다.

사용자가 로그인한 뒤 서버가 Access Token을 발급하면, 클라이언트는 이후 API 요청에 이 토큰을 포함합니다.

일반적인 전달 방식은 다음과 같습니다.

```http
GET /api/user/profile
Authorization: Bearer <access_token>
```

Access Token의 목적은 다음과 같습니다.

```text
1. 사용자가 로그인된 사용자임을 증명
2. 보호 API 접근 허용
3. 사용자 식별
4. 권한 또는 역할 확인
```

Access Token은 보통 수명이 짧습니다.

예시:

```text
5분
10분
15분
30분
1시간
```

수명을 짧게 두는 이유는 Access Token이 탈취되었을 때 피해 시간을 줄이기 위해서입니다.

---

## 5. Refresh Token이란 무엇인가

Refresh Token은 Access Token을 새로 발급받기 위한 토큰입니다.

Access Token이 만료되면 사용자가 매번 다시 로그인하지 않도록, Refresh Token을 이용해 새 Access Token을 발급받을 수 있습니다.

Refresh Token의 목적은 다음과 같습니다.

```text
1. Access Token 만료 후 새 Access Token 발급
2. 사용자 로그인 상태 연장
3. 세션 유지
4. 로그아웃 또는 강제 만료 시 재발급 차단 근거 제공
```

Refresh Token은 Access Token보다 수명이 긴 경우가 많습니다.

예시:

```text
1일
7일
14일
30일
```

단, Refresh Token은 Access Token보다 더 민감합니다.

왜냐하면 Refresh Token을 가지고 있으면 새 Access Token을 계속 발급받을 수 있기 때문입니다.

---

## 6. Access Token과 Refresh Token의 차이

| 구분       | Access Token       | Refresh Token           |
| -------- | ------------------ | ----------------------- |
| 주 용도     | API 접근             | Access Token 재발급        |
| 사용 위치    | 일반 API 요청          | Refresh API 요청          |
| 수명       | 짧음                 | 상대적으로 김                 |
| 탈취 시 영향  | 만료 전까지 API 접근 가능   | 새 Access Token 발급 가능    |
| 서버 저장 여부 | 구현에 따라 다름          | 서버 저장 또는 식별 관리 권장       |
| 폐기 정책    | 블랙리스트 또는 짧은 만료로 대응 | 로그아웃/강제만료/Rotation 등 필요 |
| 토큰 타입    | `access`           | `refresh`               |

핵심은 다음입니다.

```text
Access Token = API 접근권
Refresh Token = Access Token 재발급권
```

따라서 Access Token과 Refresh Token은 역할이 분리되어야 합니다.

---

## 7. 정상적인 토큰 발급 흐름

정상적인 로그인 흐름은 다음과 같습니다.

```text
1. 사용자가 로그인 정보를 입력한다.
2. 클라이언트가 로그인 API를 호출한다.
3. 서버가 사용자 정보를 검증한다.
4. 서버가 Access Token을 발급한다.
5. 서버가 Refresh Token을 발급한다.
6. 서버가 Refresh Token을 저장소에 저장하거나 식별 가능하게 관리한다.
7. 클라이언트는 Access Token을 API 요청에 사용한다.
8. 클라이언트는 Refresh Token을 정책에 맞게 보관한다.
```

예시:

```http
POST /api/auth/login
Content-Type: application/json

{
  "username": "test",
  "password": "1234"
}
```

응답 예시:

```json
{
  "accessToken": "<access_token>",
  "refreshToken": "<refresh_token>"
}
```

또는 Refresh Token을 Body로 내려주지 않고 HttpOnly Cookie로 내려줄 수도 있습니다.

```http
Set-Cookie: refresh_token=<refresh_token>; HttpOnly; Secure; SameSite=Lax
```

## 8. Refresh Token은 어디에 저장되는가

Refresh Token 저장 위치는 구현 방식에 따라 다릅니다.

중요한 것은 “클라이언트가 재발급 요청 시 서버에 제시할 수 있는 값이 있어야 한다”는 점입니다.

### 8.1 클라이언트 측 저장 위치

Refresh Token은 클라이언트 측에서 다음 위치에 있을 수 있습니다.

```text
1. HttpOnly Secure Cookie
2. 일반 Cookie
3. LocalStorage
4. SessionStorage
5. JavaScript Memory
6. 모바일 앱 보안 저장소
```

웹 환경에서는 보안상 HttpOnly Secure Cookie를 사용하는 경우가 많습니다.

HttpOnly Cookie를 사용하면 JavaScript에서는 Refresh Token 값을 읽을 수 없습니다.

따라서 개발자도구 콘솔이나 프론트 코드에서 Refresh Token이 보이지 않을 수 있습니다.

그러나 이 경우에도 Refresh Token은 브라우저 쿠키에 존재하며, `/refresh` 요청 시 브라우저가 자동으로 서버에 전송합니다.

즉, 다음 두 문장은 다릅니다.

```text
JavaScript에서 Refresh Token이 보이지 않는다.
클라이언트에 Refresh Token 또는 식별값이 전혀 없다.
```

첫 번째는 정상일 수 있습니다.

두 번째는 Refresh Token 기반 구조로 보기 어렵습니다.

---

### 8.2 서버 측 저장 위치

서버는 Refresh Token을 다음 위치에 저장하거나 식별할 수 있습니다.

```text
1. DB
2. Redis
3. Session Store
4. Memory
5. jti 기반 저장소
```

서버 저장의 목적은 다음과 같습니다.

```text
1. 로그아웃 시 Refresh Token 폐기
2. 강제 로그아웃
3. 토큰 탈취 대응
4. Refresh Token Rotation
5. 재사용 탐지
6. 사용자별 세션 목록 관리
7. 비정상 Refresh 요청 탐지
```

서버는 Refresh Token 원문을 그대로 저장하기보다 해시 또는 jti를 저장하는 방식도 사용할 수 있습니다.

---

## 9. “Refresh Token은 서버에만 있다”는 말의 의미

“Refresh Token은 서버에만 있다”는 표현은 정확히 구분해야 합니다.

### 9.1 가능한 의미 1: 서버에도 저장한다

다음 의미라면 정상일 수 있습니다.

```text
Refresh Token을 클라이언트에도 전달하고,
서버도 DB/Redis에 저장해서 대조한다.
```

이 구조는 일반적인 Refresh Token 관리 방식 중 하나입니다.

흐름:

```text
클라이언트가 Refresh Token을 보냄
→ 서버가 DB/Redis의 저장값과 대조
→ 유효하면 새 Access Token 발급
```

이때 “서버에 저장한다”는 말은 맞습니다.

---

### 9.2 가능한 의미 2: 클라이언트에는 없고 서버에만 있다

다음 의미라면 주의가 필요합니다.

```text
클라이언트에는 Refresh Token도, 세션 쿠키도, 식별값도 없고
서버에만 Refresh Token이 있다.
```

이 경우 서버는 Refresh 요청이 왔을 때 어떤 사용자/세션의 Access Token을 재발급해야 하는지 식별하기 어렵습니다.

클라이언트는 어떤 형태로든 서버에 식별값을 보내야 합니다.

예시:

```text
1. Refresh Token Cookie
2. Session ID Cookie
3. Request Body의 refreshToken
4. Header의 refreshToken
5. 기타 세션 식별값
```

즉, 클라이언트가 아무 값도 보내지 않는데 서버에만 Refresh Token이 있다는 설명은 인증 구조상 불명확합니다.

---

### 9.3 가능한 의미 3: 서버 세션 방식이다

실제로는 Refresh Token이 아니라 서버 세션 방식일 수 있습니다.

흐름:

```text
클라이언트는 세션 쿠키를 보냄
서버는 세션 저장소에서 사용자 상태를 확인함
서버가 새 Access Token을 발급함
```

이 경우 클라이언트가 가진 것은 Refresh Token이라기보다 세션 식별자입니다.

그래도 클라이언트는 세션 쿠키 같은 식별값을 서버로 보내야 합니다.

---

## 10. 정상적인 Refresh 흐름

Access Token이 만료되었을 때 정상 흐름은 다음과 같습니다.

```text
1. 클라이언트가 Access Token으로 보호 API 요청
2. 서버가 Access Token 만료 확인
3. 서버가 401 Unauthorized 반환
4. 클라이언트가 Refresh API 호출
5. 클라이언트가 Refresh Token 또는 세션 식별값 전달
6. 서버가 Refresh Token 서명, 만료시간, 타입 검증
7. 서버가 DB/Redis/Session Store와 대조
8. 서버가 새 Access Token 발급
9. 클라이언트가 새 Access Token 저장
10. 기존 실패 요청을 새 Access Token으로 재시도
```

요약:

```text
Access Token 만료
→ Refresh Token 검증
→ 새 Access Token 발급
→ 기존 요청 재시도
```

## 11. Refresh 요청에는 무엇이 들어가야 하는가

Refresh 요청에는 서버가 재발급 권한을 검증할 수 있는 값이 포함되어야 합니다.

가능한 방식은 다음과 같습니다.

### 11.1 Body 방식

```http
POST /api/auth/refresh
Content-Type: application/json

{
  "refreshToken": "<refresh_token>"
}
```

### 11.2 Cookie 방식

```http
POST /api/auth/refresh
Cookie: refresh_token=<refresh_token>
```

### 11.3 Header 방식

```http
POST /api/auth/refresh
X-Refresh-Token: <refresh_token>
```

### 11.4 세션 쿠키 방식

```http
POST /api/auth/refresh
Cookie: JSESSIONID=<session_id>
```

어떤 방식이든 핵심은 같습니다.

```text
서버가 검증할 수 있는 재발급 근거가 있어야 한다.
```

---

## 12. Refresh 요청에 Access Token을 그대로 넣는 구조

다음 구조는 주의가 필요합니다.

```text
1. API 요청에 사용하던 Authorization Header를 그대로 복사한다.
2. 만료된 Access Token을 /refresh 요청에 보낸다.
3. 서버가 그 Access Token을 근거로 새 Access Token을 발급한다.
```

이 구조가 위험할 수 있는 이유는 다음과 같습니다.

```text
1. Access Token과 Refresh Token의 역할 분리가 약해진다.
2. Access Token 만료시간을 짧게 둔 의미가 약해진다.
3. Access Token 탈취 시 재발급 가능성이 생길 수 있다.
4. 로그아웃 또는 강제만료 통제가 어려울 수 있다.
5. 서버가 무엇을 근거로 재발급하는지 불명확해진다.
```

다만 예외적으로 다음 구조라면 설명 가능할 수 있습니다.

```text
만료된 Access Token은 subject 추출용으로만 사용하고,
실제 재발급 권한은 서버 세션/Redis/DB에 저장된 상태로 검증한다.
```

이 경우에도 서버가 검증하는 별도 근거가 명확해야 합니다.

---

## 13. Access Token으로 Refresh가 되면 안 되는 이유

Access Token은 API 접근권입니다.

Refresh Token은 재발급권입니다.

Access Token으로 Refresh가 가능하면 Access Token이 두 역할을 모두 갖게 됩니다.

```text
Access Token = API 접근권 + 재발급권
```

이 경우 Access Token을 짧게 만료시키는 목적이 약해집니다.

예시:

```text
Access Token 만료시간: 1분
하지만 만료된 Access Token으로 계속 새 Access Token 발급 가능
```

이러면 1분 만료 정책이 실질적으로 약화될 수 있습니다.

따라서 Refresh API는 Access Token이 아니라 Refresh Token 또는 세션 식별값을 검증해야 합니다.

---

## 14. Token Type을 구분해야 하는 이유

Access Token과 Refresh Token이 모두 JWT 형식이면 서버는 두 토큰을 구분해야 합니다.

예시 claim:

```json
{
  "sub": "user-123",
  "type": "access",
  "exp": 1710000000
}
```

Refresh Token 예시:

```json
{
  "sub": "user-123",
  "type": "refresh",
  "exp": 1711000000
}
```

서버 검증 기준:

```text
보호 API 요청:
type=access만 허용

Refresh API 요청:
type=refresh만 허용
```

따라서 다음 요청은 거부되어야 합니다.

```text
Access Token을 /refresh에 제출
Refresh Token을 보호 API 접근에 사용
```

---

## 15. Refresh Token을 서버에 저장하는 이유

Refresh Token을 서버에 저장하거나 식별 관리하는 이유는 다음과 같습니다.

### 15.1 로그아웃 처리

사용자가 로그아웃하면 Refresh Token을 폐기해야 합니다.

그래야 기존 Refresh Token으로 새 Access Token을 받을 수 없습니다.

```text
로그아웃
→ 서버 저장소에서 Refresh Token 제거
→ 기존 Refresh Token으로 Refresh 불가
```

---

### 15.2 강제 로그아웃

관리자가 특정 사용자를 강제 로그아웃시킬 수 있어야 합니다.

```text
관리자 강제 로그아웃
→ 해당 사용자 Refresh Token 폐기
→ 새 Access Token 발급 차단
```

---

### 15.3 탈취 대응

Refresh Token이 탈취되면 장기간 Access Token 재발급이 가능할 수 있습니다.

서버 저장소가 있으면 해당 Refresh Token을 폐기할 수 있습니다.

---

### 15.4 Rotation 및 재사용 탐지

Refresh Token Rotation은 Refresh할 때마다 Refresh Token도 새로 발급하는 방식입니다.

흐름:

```text
기존 Refresh Token 제출
→ 새 Access Token 발급
→ 새 Refresh Token 발급
→ 기존 Refresh Token 폐기
```

만약 폐기된 기존 Refresh Token이 다시 사용되면 탈취 또는 재사용 공격으로 볼 수 있습니다.

---

## 16. Refresh Token 저장 방식별 특징

| 방식                     | 장점                  | 단점                       |
| ---------------------- | ------------------- | ------------------------ |
| HttpOnly Secure Cookie | JS 접근 차단, XSS 위험 감소 | CSRF/SameSite/CORS 고려 필요 |
| LocalStorage           | 구현 쉬움               | XSS에 취약                  |
| SessionStorage         | 탭 단위 저장 가능          | XSS에 취약, 탭 종료 시 소실       |
| Memory                 | XSS 노출면 감소          | 새로고침 시 소실                |
| 모바일 보안 저장소             | 앱 환경에 적합            | 플랫폼별 구현 필요               |
| 서버 세션                  | 통제 용이               | 서버 상태 관리 필요              |

보안상 웹에서는 Refresh Token을 HttpOnly Secure Cookie로 관리하는 방식을 많이 검토합니다.

다만 Cookie 방식은 CORS, SameSite, CSRF 정책을 함께 설계해야 합니다.

---

## 17. Refresh Token과 CORS / Cookie

프론트엔드와 백엔드 도메인이 다르면 CORS 설정이 중요합니다.

Cookie 기반 Refresh Token을 사용하는 경우 다음 설정이 필요할 수 있습니다.

클라이언트:

```javascript
axios.create({
  withCredentials: true
});
```

서버:

```text
Access-Control-Allow-Credentials: true
Access-Control-Allow-Origin: 특정 Origin
```

Cookie:

```text
HttpOnly
Secure
SameSite
Domain
Path
Expires 또는 Max-Age
```

주의할 점:

```text
Access-Control-Allow-Origin: * 와 credentials=true는 같이 사용할 수 없다.
```

따라서 Cookie 기반 Refresh Token 구조에서는 CORS와 Cookie 정책을 명확히 확인해야 합니다.

---

## 18. 동시 요청과 Refresh 중복 문제

웹 화면에서는 여러 API 요청이 동시에 발생할 수 있습니다.

예시:

```text
GET /api/menu
GET /api/user
GET /api/permission
GET /api/dashboard
GET /api/notice
GET /api/code
GET /api/config
GET /api/list
```

Access Token이 만료된 상태라면 이 요청들이 동시에 401을 받을 수 있습니다.

잘못된 구조:

```text
요청 8개가 401
→ 각 요청이 Refresh 수행
→ Refresh 요청 8회 발생
```

정상 구조:

```text
요청 8개가 401
→ Refresh 요청 1회
→ 나머지 요청은 대기
→ 새 Access Token 발급
→ 기존 요청 8개 재시도
```

이 구조를 흔히 single-flight, request queue, refresh lock 등으로 설명할 수 있습니다.

---

## 19. Refresh를 1회로 줄이는 것만으로는 부족하다

Refresh 요청을 1회로 줄였는데도 문제가 생길 수 있습니다.

예시:

```text
요청 8개가 401
→ Refresh 요청 1회
→ 최초 요청 1개만 재시도
→ 나머지 7개는 실패
```

이 구조도 정상이라고 보기 어렵습니다.

정상 구조는 다음입니다.

```text
Refresh 요청 1회
+ 실패 요청 대기열 보관
+ 새 Access Token 발급
+ 실패 요청 전체 재시도
```

즉, 핵심은 단순히 Refresh 요청 수를 줄이는 것이 아니라, 사용자 화면이 정상 복구되도록 기존 실패 요청을 재시도하는 것입니다.

---

## 20. Embedded Resource와 Refresh Token의 관계

Embedded Resource는 HTML 또는 화면 렌더링 중 함께 로드되는 하위 리소스입니다.

예시:

```text
JavaScript
CSS
Image
Font
iframe
정적 파일
첨부파일
```

Embedded Resource 때문에 요청 수가 많아질 수는 있습니다.

하지만 Embedded Resource가 많다고 해서 Refresh 요청이 요청 수만큼 발생해야 하는 것은 아닙니다.

구분해야 할 점:

```text
요청이 여러 개 발생하는 원인 = 화면 구조, Embedded Resource, API 분리
Refresh가 여러 번 발생하는 원인 = 인증 갱신 로직의 중복 실행
```

따라서 Embedded Resource는 JWT Refresh 중복을 정당화하는 근거가 아닙니다.

## 21. JWT 구조에서 반드시 확인해야 할 질문

JWT 구조를 확인할 때는 다음 질문이 중요합니다.

```text
1. Access Token의 만료시간은 얼마인가?
2. Refresh Token은 실제로 존재하는가?
3. Refresh Token은 어디에 저장되는가?
4. Refresh 요청 시 클라이언트는 어떤 값을 서버에 보내는가?
5. 그 값은 Cookie, Header, Body 중 어디에 있는가?
6. 서버는 Refresh Token을 DB/Redis/Session과 대조하는가?
7. 만료된 Access Token만으로 새 Access Token이 발급되는가?
8. Access Token과 Refresh Token의 type을 구분하는가?
9. 로그아웃 시 Refresh Token이 폐기되는가?
10. 동시 401 발생 시 Refresh 요청은 몇 번 발생하는가?
11. Refresh 성공 후 기존 실패 요청은 재시도되는가?
```

## 22. 정상 구조와 비정상 구조 비교

| 항목               | 정상 구조               | 점검 대상 구조            |
| ---------------- | ------------------- | ------------------- |
| Access Token 용도  | API 접근              | API 접근 + 재발급        |
| Refresh Token 존재 | 명확함                 | 불명확함                |
| Refresh 근거       | Refresh Token 또는 세션 | 만료 Access Token만 사용 |
| 서버 저장소 대조        | 있음                  | 없음 또는 불명확           |
| 동시 401 처리        | Refresh 1회 + 대기열    | Refresh 다중 발생       |
| 기존 실패 요청         | 새 토큰으로 재시도          | 일부만 성공 또는 실패 방치     |
| 로그아웃             | Refresh 폐기          | 기존 토큰 계속 사용 가능      |
| 보안 효과            | 만료 정책 유지            | 만료 정책 약화 가능         |

## 23. 비전문가용 비유

Access Token과 Refresh Token은 출입증과 재발급권으로 비유할 수 있습니다.

```text
Access Token = 사무실 출입증
Refresh Token = 출입증 재발급권
```

Access Token은 문을 열 때 사용하는 출입증입니다.

Refresh Token은 출입증이 만료되었을 때 새 출입증을 발급받기 위한 재발급권입니다.

문제는 다음 상황입니다.

```text
만료된 출입증을 들고 갔더니
아무 추가 확인 없이 새 출입증을 계속 발급해준다.
```

이 경우 출입증 만료 정책의 의미가 약해집니다.

정상 구조는 다음에 가깝습니다.

```text
만료된 출입증만으로는 재발급 불가
재발급권 또는 서버에 등록된 세션 확인 후 새 출입증 발급
```

---

## 24. 고객사 설명용 핵심 문장

고객사 또는 개발사에 설명할 때는 다음 문장을 사용할 수 있습니다.

```text
Refresh Token을 서버에 저장하는 것 자체는 정상일 수 있습니다.

다만 Refresh 요청 시 클라이언트가 서버에 제시하는 값이 무엇인지가 중요합니다.

그 값은 HttpOnly Cookie, Header, Body, Session ID 중 하나일 수 있습니다.

서버는 해당 값을 DB, Redis, Session Store 등과 대조하여 유효성을 검증한 뒤 새 Access Token을 발급해야 합니다.

클라이언트가 아무 값도 제시하지 않거나, 만료된 Access Token만으로 새 Access Token이 발급된다면 Access Token과 Refresh Token의 역할 분리가 불명확해질 수 있습니다.
```

---

## 25. 결론

JWT 인증 구조의 핵심은 다음입니다.

```text
Access Token은 API 접근용이다.
Refresh Token은 Access Token 재발급용이다.
두 토큰의 역할은 분리되어야 한다.

Refresh Token은 서버에도 저장될 수 있다.
그러나 Refresh 요청 시 클라이언트도 서버가 검증할 수 있는 값을 제시해야 한다.

JavaScript에서 Refresh Token이 보이지 않는 것과
클라이언트에 아무 값도 없는 것은 다른 의미다.

Access Token 만료 시 Refresh 요청은 1회로 단일화하고,
동시에 실패한 요청은 새 Access Token으로 재시도해야 한다.
```

따라서 JWT 구조 점검 시에는 “토큰이 있다/없다”보다 다음을 확인해야 합니다.

```text
1. 어떤 토큰이 어떤 용도로 쓰이는가?
2. 재발급 요청 시 서버가 무엇을 검증하는가?
3. 토큰 만료와 폐기 정책이 실제로 동작하는가?
4. 동시 인증 실패 상황에서 Refresh 요청이 중복되지 않는가?
5. Refresh 이후 기존 실패 요청이 정상 복구되는가?
```
